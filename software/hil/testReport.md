# HIL Test Report

Hardware-in-the-loop (HIL) testing on this project means the CH32V003 device-under-test (DUT) is exercised on a real bench: an **ADALM2000 (M2K)** generates stimulus (pulse trains, bursts, DC/ramp levels, a programmable supply, and byte-exact bit-banged Modbus/RS-485 frames), a **Saleae Logic16** captures the resulting digital signals through the Logic 2 application's built-in **MCP server**, the **WCH-LinkE** flashes and powers the DUT and serves as the UART monitor, and the sibling **`windmeters-modbus-interface-tester`** device acts as a well-formed Modbus master on the RS-485 bus. This report consolidates every HIL test executed to date — each with its setup, expected result, pass criteria, and pass/fail verdict grounded in the recorded numbers. The **authoritative requirement definitions (FR-Sxx, FR-MBxx, NFR-xx) live in `design/TDS.md`**; this document only cites them. The report is generated from the HIL check scripts under `software/hil/`, `design/driverDevelopment.md`, and `design/integrationPlan.md`, and **should be regenerated whenever those change**.

## Test setup

### Instruments

Three instruments make up the bench, plus the DUT and a companion master:

- **Saleae Logic16 (classic, digital-only variant) — Observer.** Captures digital signals, runs protocol analyzers, and exports CSV, driven via the Logic 2 application's built-in MCP server. It is the *classic* Logic16, digital-only: it **rejects `digitalThresholdVolts`** (its default range suits 3.3 V logic), and its **lead labels do NOT equal channel indices** (two 8-lead banks), so after any rewire signals must be located by an all-channel sweep.
- **ADALM2000 (M2K) — Stimulus generator.** Produces pulse trains, bursts, DC levels/ramps, and a programmable supply; its own scope also verifies analog. Driven via **libm2k 0.9.0**. **Firmware v0.33** is on the unit; **firmware ≥ 0.32 is MANDATORY** for libm2k 0.9.0 (the shipped fw v0.27 gave erratic AWG output and version errors). FW update method: copy `m2k.frm` onto the M2K mass-storage drive and eject — it self-flashes (~1 min, do not unplug).
- **WCH-LinkE — three jobs at once.** Flashes the DUT (`pio run -t upload`), supplies 3.3 V DUT power, and serves as the UART monitor RX. Invoked via PlatformIO's `pio.exe` at `%USERPROFILE%\.platformio\penv\Scripts\pio.exe`.
- **DUT:** a **CH32V003** (ch32v003fun toolchain).
- **Companion master:** the `windmeters-modbus-interface-tester` device also sits on the RS-485 bus as both a third bus node and a scriptable well-formed Modbus master (machine API documented in its `manual/api.md`).

### Environments

- **Saleae scripts** are stdlib-only — any Python ≥ 3.10 works (system Python 3.13 is fine).
- **libm2k** runs in a dedicated **Python 3.11 venv (`.venv-m2k`)** because libm2k wheels top out at cp311. The wheel alone is insufficient — it also needs the system-installed libm2k/libiio DLLs. One-time setup: (1) run `PlutoSDR-M2k-USB-Drivers.exe` as admin (IIO/RNDIS USB drivers); (2) run `libm2k-0.9.0-Windows-setup.exe` as admin (runtime DLLs); (3) `python3.11 -m venv .venv-m2k` then `.venv-m2k\Scripts\pip install libm2k-0.9.0-cp311-cp311-win_amd64.whl` (wheel from the libm2k GitHub release `python-wheels.zip`).
- **PlatformIO:** flashing/upload and UART monitoring go through `pio.exe` at `%USERPROFILE%\.platformio\penv\Scripts\pio.exe` (`pio run -t upload`).
- **Saleae MCP endpoint:** the Logic 2 built-in MCP server at `http://127.0.0.1:10530/mcp` (JSON-RPC over HTTP).

### Rigs

Three physical rigs are used, each reproducible from `software/hil/README.md`. All grounds (M2K, Saleae, WCH-LinkE, DUT — and both transceivers on the RS-485 rigs) are **common** throughout.

#### TTL rig (driver-development phases + integration stages 1–3)

- DUT is the CH32V003, powered at 3.3 V and flashed/monitored by the WCH-LinkE.
- The Saleae Logic16 observes DUT digital signals. Lead labels ≠ channel indices (two 8-lead banks) — after any rewire, run an all-channel sweep to locate signals. **DUT PD6 (debug UART / Modbus node) was on Saleae channel 8** during these phases.
- Driver-phase signal-verification sub-rig (`m2k_signal_check.py`): **M2K DIO0 → Saleae channel 15**; **M2K W1 → M2K scope 1+**; **M2K V+ → M2K scope 2+**.
- M2K AWG outputs are kept configured 0–3.3 V near the DUT (the hardware can swing ±5 V, beyond CH32V003 absolute maximums).
- Phase-3 / integration Modbus master: the **M2K acts as the shared-wire Modbus master in open-drain output mode** (`setOutputMode(DIO_OPENDRAIN)` + external pull-up), **permanently enabled** — the tristate direction-flip-per-transaction approach sprayed break glitches and phantom start bits and was rejected.

#### MAX3485 transceiver rig (2026-07-06 — `rs485_check.py` / `rs485_regs_check.py`)

- The DUT drives a MAX3485 transceiver. **DUT DI+RO (transceiver data-in / receiver-out node) → DUT PD6, observed on Saleae channel 8** on that node.
- **DUT DE+/R̄Ē (driver-enable / receiver-enable-bar, tied) → DUT PC2, observed on Saleae channel 15, with a 10 kΩ pull-down.**
- The **RS-485 bus A/B lines → M2K scope 1+ / 2+** for the analog wire view (optionally also Saleae ch0/ch1).
- The `windmeters-modbus-interface-tester` is the live well-formed master on the bus (driven through its HTTP machine API, `POST /api/v1/modbus`, which returns raw TX/RX frames for independent CRC verification); the DUT is the passive judge / responder.

#### Second-MAX3485 M2K-driven raw master (`rs485_raw_check.py`)

- A **second MAX3485** transceiver, driven directly by the M2K, injects byte-exact / malformed frames as a raw master, sharing the same RS-485 bus as the DUT and the tester (**three nodes total**).
- Wiring: **M2K DIO0 → DI** (data in); **M2K DIO1 → DE+/R̄Ē** (driver-enable, tied); **M2K V+ → VCC (3.3 V)** of that second MAX3485.
- Every raw-master script MUST enable V+ itself (`ps.enableChannel(0, True)` + `pushChannel(0, 3.3)`) — an unpowered MAX3485 sits inert. The static DE/DI test (drive space, drive mark, release) is the 10-second wiring proof.

### Bench quirks that change how results must be read

- **Analyzer unreliable at 9600 baud.** The Logic 2 Async Serial analyzer scrambles byte values at 9600 baud on this setup while the raw edges are perfect. Binary/Modbus captures are therefore decoded with `saleae_serial.uart_decode()` over the raw-edge export, using `decode_events(..., sync_to_newline=False)` for binary (the text first-LF rule would silently discard LF-free captures). Accuracy-of-record for `wd_check.py` uses a **resistor divider**, not the analyzer/AWG.
- **M2K analog-out session state is fragile.** Within one context, repeated output reconfiguration misbehaves (`stop()` wedges the DAC until `reset()`; a second cyclic `push()` at the same rate is silently ignored; stale state leaks). The reliable pattern is a **fresh `m2kOpen()` + calibrate per analog stimulus configuration** (~1.5 s each; digital out is unaffected). The libm2k context MUST stay OPEN while the DUT-side capture runs — `contextClose()` idles the AWG. Rapid close/reopen can transiently fail ("Cannot set the number of kernel buffers") — retry with ~1 s backoff. First pattern-generator period after `push()` is anomalous (judge steady state); large cyclic digital buffers get truncated (scale to ~10k samples/period); AWG sample rates snap to a 75 MS/s/10ⁿ ladder and DC droops between wraps (generate DC at 750 kS/s or via `setVoltage()`). A "disabled" AWG channel is NOT high-impedance (~50 Ω to idle) so it cannot emulate a disconnected sensor — physically lift the wire.
- **M2K absolute-voltage accuracy is not trustworthy.** DMM-anchored, the AWG outputs setpoint +25 mV (constant offset) and the scope reads ~1 %/−30 mV low (affine, drifts ~10 mV between sessions). **Never assert DUT accuracy against M2K absolute voltages** — use a resistor divider from DUT VDD (ratiometric, DMM-measured ratio) as the method of record. The M2K stays fine for dynamics, end stops, and anything ratio-cancelled.
- **Tester buffers overheard bus traffic.** The tester never stops listening; its Modbus master only drains its UART RX during its own transactions, so it buffers every byte the raw master puts on the shared bus, then parses the stale backlog on its next transaction and reports CRC_ERR (a log showed ~30 overheard frames ahead of the true response). **One throwaway read IS the flush** — retry once on `crc_error` before trusting a tester-API read after raw-master traffic (`rs485_raw_check.dut_diag`).
- **Noise floods take the silent-discard path.** Random-byte floods do NOT move the DUT's CRC counter (register 30009) — those bytes arrive framing-poisoned (FE/overflow) and are silently discarded; only cleanly-framed-wrong-CRC frames (e.g. split halves) increment the counter. **Assert flood recovery by behavior, not by that counter.** Inter-frame marks in composed patterns must exceed **t3.5 (4.01 ms)** or the DUT correctly coalesces frames into one discard — the house gap is 48 bit times of mark (5 ms); a 2.5 ms gap between a garbage tail and a recovery request merged them.
- **Saleae MCP / capture quirks.** `add_analyzer` settings need tagged values (`{"Input Channel": {"numberValue": 8}}`); `export_raw_data_csv` requires `analogDownsampleRatio` even for digital-only; the Async Serial data-table CSV holds the literal character per byte (CR/LF arrive as embedded newlines inside quoted fields — don't strip); a capture starting mid-byte yields garbage + one framing error before the first line boundary (judge from the first clean newline onward); the classic Logic16 rejects `digitalThresholdVolts` (omit it).
- **Baud tolerance.** The DUT answers master baud offsets to ±3 % both directions (measured against the ladder read-back rate); its own HSI adds ~+0.3 %.
- **DUT-side UART / timing gotchas.** HDSEL half-duplex idles the line floating (framing errors on bursty output) so `common/debug_uart` runs plain TX with idle mark driven; HDSEL also intermittently swallows the first RX byte after idle (~35 %, no error flags), so the Modbus driver abandoned HDSEL for a **RX-on-PD6-default-map / TX-remapped-in-for-response** discipline. Interrupts are suspect on this toolchain path (an RXNE ISR corrupted ~1/3 of frames with no USART error flags; **polled RX** fixed it — the architecture is now zero-ISR). A master must release the wire immediately after its last stop bit (the DUT replies ~5 ms later; holding the line collides).
- **Timing-scale gotcha.** ch32v003fun SysTick defaults to HCLK/8 — define **`FUNCONF_SYSTICK_USE_HCLK 1`** for raw `SysTick->CNT` pacing or everything runs 8× slow. Count-vs-frequency asserts must scale to the **MEASURED window duration** (report-line timestamp deltas), because the DUT window is HSI-paced (±1 %) — a nominal-window compare at 1 kHz is off by several counts while the DUT behaves exactly per FR-S06.
